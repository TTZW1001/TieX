# 七、PathGuard 要求

## 7.1 安全设计

PathGuard 是工作区受限沙箱的核心安全模块，负责所有文件路径的安全校验。

## 7.2 路径校验流程

```text
接收相对路径
  ↓
拒绝空路径和非法字符（\0、..\0 等）
  ↓
拒绝绝对路径（以 / 或盘符开头）
  ↓
拒绝 UNC 网络路径（以 \\ 开头）
  ↓
拒绝包含 .. 的路径（防止遍历）
  ↓
path.resolve(workspaceRoot, relativePath)
  ↓
获取真实路径 realpath.native（解析符号链接）
  ↓
检查真实路径是否仍在工作区内（path.relative 判断）
  ↓
检查路径对应的文件或父目录是否存在
  ↓
允许访问
```

## 7.3 核心方法

```ts
interface IPathGuard {
  /**
   * 校验相对路径是否安全
   * 返回 { allowed: true } 或 { allowed: false, reason: string }
   */
  validate(workspaceRoot: string, relativePath: string): PathValidationResult

  /**
   * 校验路径是否在工作区内
   */
  isWithinWorkspace(workspaceRoot: string, targetPath: string): boolean

  /**
   * 获取安全的绝对路径（校验通过后返回）
   */
  resolveSafePath(workspaceRoot: string, relativePath: string): string | null

  /**
   * 检查文件类型是否允许读取
   * 默认只允许文本文件，二进制文件拒绝
   */
  isReadableFile(filePath: string): boolean

  /**
   * 检查文件大小是否在限制内
   */
  isWithinSizeLimit(filePath: string, maxBytes: number): boolean
}

interface PathValidationResult {
  allowed: boolean
  resolvedPath?: string
  reason?: string
}
```

## 7.4 拒绝场景

必须拒绝：

- 空路径或空白路径；
- 非法字符路径（含 null 字符）；
- 绝对路径（如 `C:\Windows\system32`、`/etc/passwd`）；
- UNC 网络路径（如 `\\server\share`）；
- 路径遍历（如 `../../etc/passwd`、`foo/../../../bar`）；
- 符号链接指向工作区外（通过 `fs.realpath.native` 解析后校验）；
- 工作区路径本身不存在时拒绝所有操作；
- 目标文件的父目录不存在（新文件场景预留）。

## 7.5 文件读取限制

- 默认仅读取文本文件（扩展名白名单：`.ts`、`.js`、`.json`、`.md`、`.txt`、`.css`、`.html`、`.vue`、`.yaml`、`.yml`、`.toml`、`.xml`、`.env`、`.gitignore`、`.editorconfig`、`.py`、`.java`、`.rs`、`.go`、`.c`、`.cpp`、`.h`、`.sql`、`.sh`、`.bat`、`.ps1`）；
- 单文件大小默认限制 1MB；
- 超大文件可分段读取（默认每段 64KB）；
- 二进制文件默认拒绝；
- 默认忽略 `node_modules/` 和 `.git/` 目录；
- 不得读取工作区外的链接目标。

# 九、只读工具要求

## 9.1 list_files

列出工作区指定路径下的文件和子目录。

### 参数

```ts
interface ListFilesInput {
  path: string           // 工作区内的相对路径，默认为 ''
  includeHidden?: boolean // 是否包含隐藏文件，默认 false
  maxDepth?: number       // 递归深度，0 表示仅当前目录，默认 1
}
```

### 返回

```ts
interface FileEntry {
  name: string
  path: string           // 相对路径
  type: 'file' | 'directory'
  size: number           // 文件大小，目录为 0
  extension: string      // 文件扩展名
  modifiedAt: string     // 最后修改时间
}
```

### 规则

- `path` 参数必须经过 PathGuard 校验；
- 默认忽略 `node_modules/`、`.git/` 和 `.gitignore` 中匹配的目录；
- 不列出超过 1000 个条目的目录（防止超大目录卡死）；
- `maxDepth` 最大允许 3。

## 9.2 read_file

读取工作区内的文本文件内容，支持分段读取。

### 参数

```ts
interface ReadFileInput {
  path: string            // 工作区内的相对路径
  startOffset?: number    // 起始偏移量（字节），默认 0
  maxLength?: number      // 最大读取长度（字节），默认 65536（64KB）
}
```

### 返回

```ts
interface ReadFileResult {
  content: string
  totalSize: number       // 文件总大小
  startOffset: number     // 实际起始偏移
  endOffset: number       // 实际结束偏移（content 的结束位置）
  isTruncated: boolean    // 是否被截断（文件过长时）
}
```

### 规则

- `path` 参数必须经过 PathGuard 校验；
- 文件大小超过 `maxLength` 时截断返回并标记 `isTruncated: true`；
- 二进制文件拒绝读取；
- 每次最大读取 1MB，超出部分需分段获取。

## 9.3 search_files

在工作区内搜索匹配指定 pattern 的文件或内容。

### 参数

```ts
interface SearchFilesInput {
  pattern: string          // 搜索关键词或正则
  path?: string            // 搜索起始路径，默认工作区根目录
  filePattern?: string     // 文件匹配模式，如 '*.ts'、'*.json'
  maxResults?: number      // 最大返回结果数，默认 50
  searchContent?: boolean  // 是否搜索文件内容，默认 false
}
```

### 返回

```ts
interface SearchResult {
  path: string             // 相对路径
  fileName: string
  matches?: SearchMatch[]  // 内容匹配详情（仅 searchContent=true 时）
}

interface SearchMatch {
  line: number
  column: number
  content: string          // 匹配行内容（最多 200 字符）
}
```

### 规则

- 使用 `fast-glob` 或 Node.js `fs` 实现文件匹配；
- 搜索内容时使用 Node.js 流式读取，避免大文件卡死；
- 默认忽略 `node_modules/`、`.git/` 和二进制文件；
- 每个文件最多返回 10 个匹配行；
- `maxResults` 上限 200。

```bash
npm install fast-glob
```

不得引入当前阶段不需要的依赖，如：

- diff
- docx
- pptxgenjs
- zod 或 ajv
- monaco-editor

------

# 十四、本阶段禁止内容

本阶段绝对禁止实现：

- Agent Runtime；
- Tool Calls 与模型联动（工具仅通过 IPC 直接调用，不经过 Agent）；
- 文件创建和修改（create_file、edit_file）；
- 文件删除；
- DOCX 生成；
- PPTX 生成；
- 命令执行；
- Git 操作；
- 权限审批真实逻辑（PermissionService 只预留接口）；
- 文件备份与恢复；
- 多模型切换；
- 插件；
- MCP；
- 自动化；
- 多智能体；
- 用户登录；
- 云同步。

即使为了"预留接口"，也不要建立可以直接执行文件或命令的通用 IPC。

# 十五、代码质量要求

1. 使用 TypeScript，避免滥用 `any`；
2. PathGuard 必须在所有文件操作前执行校验；
3. 文件读取必须处理编码问题（使用 `utf-8`）；
4. IPC 处理函数必须校验参数；
5. 文件搜索结果必须限制返回数量；
6. 大文件读取必须分段，防止内存溢出；
7. 工作区信息写入数据库使用参数化查询；
8. 文件树组件使用懒加载，避免一次性加载所有文件；
9. 不破坏阶段一、阶段二、阶段三已有的界面功能；
10. 不修改文档内容；
11. 不修改原型文件。
