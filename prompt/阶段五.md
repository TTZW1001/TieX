# 七、Agent Runtime 要求

## 7.1 总体流程

```
用户输入任务
  ↓
创建 Task（状态：pending）
  ↓
初始化 Agent Runtime
  ↓
构建上下文（系统提示词 + 历史 + 工具列表）
  ↓
调用 DeepSeek（streamChat，传入工具定义）
  ↓
模型返回 → Response Parser 解析
  ├─ final_text → 保存 Assistant 消息，标记任务完成
  ├─ tool_calls → 逐个处理工具调用
  └─ 非法响应 → 错误回传，增加失败计数
  ↓
处理 Tool Call
  ↓
Tool Executor 校验并执行
  ↓
结果回传模型
  ↓
继续下一轮（循环）
  ↓
达到限制 / 用户停止 / 出错 → 任务结束
```

## 7.5 Agent Loop 伪代码

```ts
async function runAgent(context: RuntimeContext): Promise<void> {
  updateTaskStatus(context.taskId, 'running')

  while (true) {
    // 检查是否被停止
    if (context.abortController.signal.aborted) {
      await saveAssistantMessage('任务已被用户停止')
      await completeTask(context.taskId, 'stopped')
      return
    }

    // 检查任务限制
    checkWithinLimits(context)

    // 构建上下文
    const modelRequest = await contextBuilder.build(context)

    // 调用模型
    const modelResult = await provider.streamChat(
      modelRequest,
      context.abortController.signal
    )

    context.round += 1

    if (modelResult.type === 'final_text') {
      await saveAssistantMessage(modelResult.content)
      await completeTask(context.taskId, 'completed')
      return
    }

    if (modelResult.type === 'tool_calls') {
      for (const toolCall of modelResult.toolCalls) {
        if (context.abortController.signal.aborted) break
        context.toolCallCount += 1
        checkWithinLimits(context)
        await handleToolCall(context, toolCall)
      }
      continue  // 继续下一轮
    }

    // 非法响应
    context.failureCount += 1
    checkWithinLimits(context)
    await pushToolError(context, 'MODEL_RESPONSE_INVALID', '模型返回格式异常')
  }
}
```

## 7.6 TaskLimits

```ts
interface TaskLimits {
  maxModelRounds: number        // 默认 20
  maxToolCalls: number          // 默认 30
  maxFailures: number           // 默认 3
  maxDurationMs: number         // 默认 30 分钟
  maxFilesRead: number          // 默认 100
  maxFileSizeBytes: number      // 默认 2MB
  maxChangedFiles: number       // 默认 20（阶段六使用）
  maxToolOutputChars: number    // 默认 50000
}
```

超过限制后：
- 停止当前任务
- 保存错误代码
- 向用户显示原因
- 不再调用后续工具

## 7.7 一轮定义

一次完整模型请求记为一轮。

一轮可能返回：
- 最终文本（final_text）
- 一个或多个工具调用（tool_calls）
- 非法响应
- 空响应

## 7.8 多工具调用

V1.0 默认按顺序执行多个工具调用，不并行执行。

原因：
- 更容易控制权限
- 更容易保证文件顺序
- 更容易停止
- 更容易记录

# 九、Context Builder 要求

## 9.1 上下文组成

模型请求上下文包含：

1. 系统提示词（System Prompt）
2. 当前会话历史
3. 当前任务用户消息
4. 当前工作区信息
5. 当前权限模式
6. 工具列表（AgentTool 的 name、description、schema）
7. 工具调用历史（上一轮工具结果）
8. 当前任务限制信息（可选）
9. 必要文件片段（可选）

## 9.2 系统提示词

系统提示词必须明确：

- TieX 是本地桌面 Agent
- 只能通过已注册工具操作本地资源
- 工具列表与 JSON Schema
- 不得编造工具执行结果
- 不得请求未注册工具
- 不得访问工作区外路径
- 工具失败后应分析错误
- 达到任务目标后应输出最终答复
- 不应无限重复同一工具
- 当前权限模式限制

系统提示词模板保存在 `electron/main/agent/system-prompt.ts` 中。

## 9.3 会话历史裁剪

建议策略：
1. 保留系统提示词
2. 保留当前任务用户消息
3. 保留最近若干轮（如最近 10 轮）
4. 保留未完成的工具调用
5. 超出上下文时移除最早的内容

## 9.4 工作区摘要

仅包含：
- 工作区名称
- 根目录名称
- 权限模式
- 已知项目类型（可选）
- 必要目录结构摘要（可选）

不发送绝对路径，除非工具执行所需。

# 十、Response Parser 要求

## 10.1 模型响应类型

```ts
type ModelResponse =
  | { type: 'final_text'; content: string }
  | { type: 'tool_calls'; toolCalls: ModelToolCall[] }
  | { type: 'invalid'; reason: string }
```

## 10.2 ModelToolCall

```ts
interface ModelToolCall {
  id: string
  name: string
  arguments: unknown  // JSON object
}
```

## 10.3 解析规则

- deepseek-chat 模型的工具调用通过 `choices[0].delta.tool_calls` 返回
- 需要累积 delta 中的 `tool_calls` 片段（index 分组）
- 当完整收到一个 tool_call 后，解析 `arguments` 为 JSON
- 同一流中可能先返回文本内容再返回工具调用
- 先返回的文本内容可作为最终回复的补充

## 10.4 非法响应

包括：
- 工具名称为空
- 参数不是合法 JSON
- 未注册工具
- 同一调用 ID 重复执行
- 最终文本和工具调用格式冲突
- 模型返回空内容

非法响应不得执行，需将错误回传模型。

## 11.1 工具定义

```ts
interface AgentTool<TInput = unknown, TOutput = unknown> {
  name: string
  description: string
  schema: JSONSchema

```

```ts

```

## 11.4 查找

模型请求工具时：
```ts
const tool = toolRegistry.get(toolCall.name)
```

未注册工具的处理：
- 不执行
- 保存失败记录（operation_logs）
- 将错误返回模型
- 增加失败计数

## 11.5 工具风险等级

```ts
type ToolRiskLevel = 'low' | 'medium' | 'high' | 'blocked'
```

- low：list_files、read_file、search_files
- medium：创建新文件（预留）
- high：覆盖文件、修改文件、执行命令（预留）
- blocked：工作区外访问、系统目录修改（预留）

## 11.6 Schema Validator

必须实现 JSON Schema 校验：
- 所有工具参数必须通过 JSON Schema 校验
- 不接受额外未知字段（`additionalProperties: false`）
- 校验失败时返回结构化错误

```ts
interface ToolValidationError {
  code: 'TOOL_ARGUMENT_INVALID'
  toolName: string
  issues: Array<{
    path: string
    message: string
  }>
}
```

校验失败处理：
1. 不执行工具
2. 保存失败状态
3. 将简化错误返回模型
4. 允许模型修正
5. 超过失败限制后停止任务

# 十二、Tool Executor 要求

## 12.1 执行流程

```
接收 ModelToolCall
  ↓
查找工具（Registry.get）
  ↓
创建 tool_call 记录
  ↓
Schema 校验（validate）
  ↓

  ↓
风险评估（riskLevel）
  ↓

  ↓
更新 task_step 状态为 executing_tool
  ↓
执行工具
  ↓
截断和脱敏结果
  ↓
保存结果
  ↓
回传模型（作为下一轮上下文的一部分）
```

## 12.2 ToolExecutionContext

```ts
interface ToolExecutionContext {
  taskId: string
  conversationId: string
  workspaceRoot?: string
  permissionMode: PermissionMode
  signal: AbortSignal
}
```

## 12.3 结果回传格式

成功：
```json
{
  "ok": true,
  "tool": "read_file",
  "result": { ... }
}
```

失败：
```json
{
  "ok": false,
  "tool": "read_file",

    "code": "FILE_NOT_FOUND",
```

## 12.4 结果截断

- 明确 `truncated: true`

- 不因截断而假装完整成功

## 14.2 任务事件推送

```ts
type TaskEvent =
  | { type: 'task:started'; taskId: string }
  | { type: 'task:statusChanged'; taskId: string; status: TaskStatus }
  | { type: 'tool:started'; taskId: string; toolCallId: string; toolName: string }
  | { type: 'tool:completed'; taskId: string; toolCallId: string; result: unknown }
  | { type: 'tool:failed'; taskId: string; toolCallId: string; error: string }
  | { type: 'message:delta'; taskId: string; content: string; delta: string }
  | { type: 'task:completed'; taskId: string }
  | { type: 'task:failed'; taskId: string; error: string }
  | { type: 'task:stopped'; taskId: string }
```

使用 `webContents.send()` 从主进程推送到渲染进程。

## 14.3 Preload API

```ts
task: {
  create(input: CreateTaskInput): Promise<TaskInfo>

  getSteps(taskId: string): Promise<TaskStep[]>
  getToolCalls(taskId: string): Promise<ToolCall[]>
  getLogs(taskId: string): Promise<OperationLog[]>
  onEvent(callback: (event: TaskEvent) => void): () => void
}
```

- 渲染进程不得直接调用 DeepSeek API（仍由主进程发起）
- 渲染进程不得接触 API Key 明文
- 所有 Agent 逻辑在主进程执行

# 十七、DeepSeek 工具调用适配

## 17.1 请求格式

向 DeepSeek API 发送聊天请求时，需在请求体中包含 `tools` 参数：

```json
{
  "model": "deepseek-chat",
  "messages": [...],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "list_files",
        "description": "...",
        "parameters": { ... }
      }
    }
  ],
  "tool_choice": "auto"
}
```

## 17.2 流式响应解析

DeepSeek 流式响应中，工具调用通过 `choices[0].delta.tool_calls` 返回：

- 需要按 `index` 分组累积 delta
- 每个 index 包含 `function.name` 和 `function.arguments`（片段）
- 当收到 `finish_reason: "tool_calls"` 时，工具调用完整
- `finish_reason: "stop"` 表示最终文本

## 17.3 Provider 适配

阶段三的 DeepSeekProvider 需要扩展支持 `tools` 参数和 `tool_choice`。

建议在 `streamChat` 方法中增加可选的 `tools` 参数：

```ts
streamChat(
  request: ModelRequest,
  signal: AbortSignal,
  tools?: ToolDefinition[]  // 新增参数
): AsyncIterable<ModelStreamEvent>
```

完成开发后，不要只回复"已完成"。

必须按以下格式汇报：

## 1. 本阶段完成内容

逐项列出完成的 Agent Runtime、Tool Registry、数据库、IPC、前端组件等。

## 2. 新增文件

列出所有新增文件。

## 3. 修改文件

列出所有修改文件。

## 4. 安装依赖

列出新增依赖及用途。

## 5. 运行方式

给出真实可执行的安装、启动和构建命令。

## 6. 测试与构建结果

至少说明：

- TypeScript 类型检查结果；
- Vite 构建结果；
- 任务创建验证；
- Agent 工具调用验证；
- 任务停止验证。

## 7. 验收标准逐条结果

按照 P5-A01 至 P5-A20 逐条标记：

```text
✅ 通过
❌ 未通过
⚠️ 部分通过
```

并写明依据。

## 8. 当前已知问题

如果没有，也明确写：

```text
无阻塞性已知问题
```

## 9. 当前未实现内容

明确说明阶段六及以后仍未实现。

## 10. 是否建议进入下一阶段

只给出建议，不得自行开始下一阶段。

# 二十五、最终约束

1. 先阅读，再开发；
2. 先列计划，再修改；
3. 只做阶段五；
4. 不进入阶段六；
5. 不实现文件写入和修改；
6. 不实现命令执行；
7. 不实现文档/PPT 生成；
8. 不改变现有需求范围；
9. 不破坏现有文档；
10. 不破坏阶段一至阶段四已有界面功能；
11. 完成后等待我验收。

请现在先检查项目目录、阅读全部文档和现有代码，列出阶段五实施计划，然后开始开发。
