# 七、权限审批要求

## 7.1 权限模式

阶段六引入新的权限模式：

```ts
type PermissionMode = 'chat' | 'read' | 'execute'
```

- `chat`：仅普通对话，不可调用文件工具
- `read`：可调用 list_files、read_file、search_files 等只读工具
- `execute`：可执行写入工具，但高风险操作仍需审批

当用户设置了工作区时，默认使用 `execute` 模式（阶段四为 `read`，阶段六升级为 `execute`）。

## 7.2 审批流程

```text
工具请求（create_file / edit_file）
  ↓
Tool Executor 检查
  ├─ 风险 low（只读工具） → 自动执行，无需审批
  ├─ 风险 medium（创建新文件） → 需要审批
  └─ 风险 high（覆盖文件 / 修改文件） → 需要审批
  ↓
需要审批时：
  ↓
创建 permission_request（status: pending）
  ↓
更新 task 状态为 waiting_permission
  ↓
通知渲染进程弹窗
  ↓
用户选择
  ├─ 拒绝 → permission_request.status = 'rejected'，任务回退到 running
  └─ 允许一次 → permission_request.status = 'approved'，decision_scope = 'once'
  └─ 本任务允许 → permission_request.status = 'approved'，decision_scope = 'task'
  ↓
根据决策执行或跳过工具
  ↓
任务继续下一轮
```

## 7.3 PermissionService 接口

```ts
interface PermissionService {
  /**
   * 请求权限
   */
  requestPermission(input: PermissionRequestInput): Promise<PermissionRequestEntity>

  /**
   * 处理审批决策
   */
  handleDecision(requestId: string, decision: PermissionDecision): Promise<void>

  /**
   * 检查工具调用是否需要权限审批
   */
  requiresPermission(toolName: string, input: unknown): PermissionCheckResult

  /**
   * 检查任务是否已有授权
   */
  hasTaskApproval(taskId: string, toolName: string, target: string): boolean
}

interface PermissionRequestInput {
  taskId: string
  toolCallId: string
  permissionType: string
  title: string
  reason?: string
  target?: string
  impactSummary?: string
}

interface PermissionCheckResult {
  required: boolean
  reason?: string
  riskLevel?: ToolRiskLevel
}

type PermissionDecision = 'approved_once' | 'approved_for_task' | 'rejected'
```

## 7.4 PermissionPolicy

必须实现以下规则：

| 工具 | 场景 | 风险等级 | 是否需要审批 |
|------|------|----------|:--------:|
| create_file | 目标文件不存在（创建新文件） | medium | 是 |
| create_file | 目标文件已存在（覆盖） | high | 是 |
| edit_file | 修改已有文件 | high | 是 |
| list_files | 列出目录 | low | 否 |
| read_file | 读取文件 | low | 否 |
| search_files | 搜索文件 | low | 否 |

规则说明：

- 任何写入操作（create_file、edit_file）至少需要 medium 风险审批；
- 覆盖已有文件（create_file 且文件已存在）为 high 风险；
- 只读工具（list_files、read_file、search_files）为 low 风险，自动放行；
- V1.0 不实现文件删除工具；
- V1.0 不实现永久授权（仅 once 和 task）；
- 审批记录关联到 tool_call，即使同一任务，不同工具调用需单独审批。

## 7.5 任务状态

阶段六新增任务状态：

```ts
type TaskStatus =
  | 'pending'
  | 'running'
  | 'waiting_permission'   // 新增：等待用户审批
  | 'executing_tool'
  | 'completed'
  | 'failed'
  | 'stopped'
  | 'interrupted'
```

新增状态转换：

```
running → waiting_permission（需要审批时）
waiting_permission → running（用户拒绝/授权后继续执行工具）
waiting_permission → stopped（用户停止任务）
```

## 7.6 审批事件

新增事件类型：

```ts
// 主进程推送审批请求到渲染进程
{ type: 'permission:requested', taskId, requestId, toolName, title, reason, target, impactSummary }

// 渲染进程发送审批结果回主进程
IPC: 'permission:decide', { requestId, decision: 'approved_once' | 'approved_for_task' | 'rejected' }

// 审批已处理事件
{ type: 'permission:decided', taskId, requestId, decision }
```

## 8.1 create_file 工具

### 用途

创建新文本文件。

### 注册名称

```text
create_file
```

### 输入

```ts
interface CreateFileInput {
  path: string            // 工作区内的相对路径
  content: string         // 文件内容
  overwrite?: boolean     // 是否允许覆盖，默认 false
}
```

### 输出

```ts
interface CreateFileOutput {
  path: string
  sizeBytes: number
  created: boolean        // true=新建, false=覆盖
}
```

### 规则

- `path` 必须经过 PathGuard 校验；
- 默认 `overwrite = false`，文件已存在时返回错误（需设置 overwrite=true 并重新申请）；
- 父目录不存在时自动创建；
- 写入采用临时文件 + 原子替换（先写入 `.tmp` 文件再重命名）；
- 新建文件风险 medium，覆盖文件风险 high；
- 只支持文本文件（扩展名白名单同阶段四 read_file）；
- 写入前计算备份（新文件无需备份，覆盖时需要）；
- 写入后记录 file_change。

### Schema

```json
{

  "properties": {
    "path": {
      "type": "string",
      "description": "工作区内的相对路径"
    },
    "content": {
      "type": "string",
      "description": "文件内容"
    },
    "overwrite": {
      "type": "boolean",
      "description": "是否允许覆盖已有文件，默认 false"
    }

  "required": ["path", "content"],
  "additionalProperties": false
}
```

## 8.2 edit_file 工具

### 用途

修改已有文本文件。基于原文匹配的局部替换，不允许模型直接提交任意补丁脚本。

### 注册名称

```text
edit_file
```

### 输入

```ts
interface EditFileInput {
  path: string                   // 工作区内的相对路径
  edits: Array<{
    oldText: string              // 原文片段（必须匹配）
    newText: string              // 替换后的内容
  }>
}
```

### 输出

```ts
interface EditFileOutput {
  path: string
  replacements: number           // 实际替换次数
  beforeHash: string             // 修改前 SHA256
  afterHash: string              // 修改后 SHA256
  diffSummary: string            // 差异摘要（统一 diff 格式）
}
```

### 规则

- `path` 必须经过 PathGuard 校验；
- 文件必须存在；
- `oldText` 必须在文件内容中出现至少一次；
- 如果 `oldText` 匹配次数为 0，拒绝修改并返回错误；
- 如果 `oldText` 匹配次数不等于 1 且 `replaceAll` 未设置，拒绝修改；
- 写入前必须备份原文件；
- 写入前计算旧文件哈希（SHA256）；
- 写入采用临时文件 + 原子替换；
- 写入后计算新文件哈希；
- 写入后计算 Diff（统一 diff 格式）；
- 修改文件数受任务上限限制（默认 20 个）；
- 必须审批（high 风险）。

### Schema

```json
{
  "type": "object",
  "properties": {
    "path": {
      "type": "string",
      "description": "工作区内的相对路径"
    },
    "edits": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "oldText": {
            "type": "string",
            "description": "原文片段（必须在文件内容中出现）"
          },
          "newText": {
            "type": "string",
            "description": "替换后的内容"
          }
        },
        "required": ["oldText", "newText"],
        "additionalProperties": false
      },
      "minItems": 1,
      "maxItems": 10,
      "description": "编辑操作列表，最多 10 个"
    }
  },
  "required": ["path", "edits"],
  "additionalProperties": false
}
```

## 8.3 备份服务（BackupService）

### 备份目录

```text
%APPDATA%/TieX/backups/{taskId}/
```

备份文件保持原始相对目录结构。

### BackupService 接口

```ts
interface BackupService {
  /**
   * 创建文件备份，返回备份路径
   */
  createBackup(taskId: string, workspaceRoot: string, relativePath: string): Promise<string>

  /**
   * 从备份恢复文件
   */
  restoreFile(taskId: string, workspaceRoot: string, relativePath: string, backupPath: string): Promise<void>

  /**
   * 清理任务的所有备份
   */
  cleanupTaskBackups(taskId: string): void
}
```

### 备份流程

1. 读取原文件内容；
2. 在 `%APPDATA%/TieX/backups/{taskId}/` 下创建相对目录结构；
3. 将原文件复制到备份目录；
4. 返回备份文件的绝对路径；
5. 记录到 `file_changes.backup_path`。

### 恢复流程

1. 检查目标文件当前哈希；
2. 如果当前哈希与 `afterHash` 不一致（文件已被用户再次修改），提示冲突；
3. 用户确认后再覆盖；
4. 恢复后更新 `file_changes.reverted_at`；
5. 记录恢复日志。

## 8.4 文件写入安全规则

- 不得直接写入目标文件；
- 先写入同目录下的临时文件（如 `.tiex.tmp`）；
- 写入完成后使用 `fs.renameSync()` 原子替换；
- 写入失败时删除临时文件；
- 不遗留 `.tmp` 文件；
- 写入完成后校验文件哈希。

## 8.5 文件变更记录（file_changes）

操作类型枚举：

```text
create
modify
```

V1.0 不支持 `delete` 和 `move`。

status 枚举：

```text
applied
reverted
```

修改流程必须记录：

1. `operation`（create / modify）；
2. `backup_path`（修改前备份路径）；
3. `before_hash`（修改前文件 SHA256）；
4. `after_hash`（修改后文件 SHA256）；
5. `diff_summary`（统一 diff 格式的摘要）；
6. `changed_at`（修改时间）；
7. 恢复时更新 `reverted_at`。

```
查找工具（Registry.get）

风险评估（riskLevel）
  ├─ low → 继续执行
  ├─ medium/high → 进入权限审批流程
  └─ blocked → 直接拒绝
  ↓
权限审批流程（如需）
  ├─ 创建 permission_request（pending）
  ├─ 更新 task 状态为 waiting_permission
  ├─ 推送 permission:requested 事件
  ├─ 等待用户决策
  │   ├─ approved_once → 执行一次
  │   ├─ approved_for_task → 本任务后续自动放行
  │   └─ rejected → 跳过工具，返回错误
  └─ 继续执行或跳过
  ↓
工作区与路径校验（PathGuard）
  ↓
执行工具
  ↓
截断和脱敏结果
  ↓
保存结果
  ↓
记录 file_change（写入工具）
  ↓
回传模型
```

## 9.2 审批等待机制

- Tool Executor 遇到需要审批的工具时，不直接拒绝也不直接执行；
- 创建 `permission_request` 记录；
- 更新任务状态为 `waiting_permission`；
- 通过事件总线推送 `permission:requested` 事件到渲染进程；
- 等待用户决策（通过 IPC `permission:decide` 接收）；
- 用户决策后根据结果执行或跳过；
- 等待期间任务可以被用户停止；
- 任务被停止时，所有 pending 的 permission_request 标记为 cancelled。

# 十七、验收标准

| 验收项 | 说明 |
|--------|------|
| 新文件可创建 | Agent 可通过 create_file 在工作区内创建新文本文件 |
| 已存在文件不会静默覆盖 | 覆盖需 overwrite=true 并经过审批 |
| 修改文件前产生审批 | edit_file 执行前弹出审批弹窗 |
| 拒绝后不执行 | 用户拒绝后工具跳过，错误回传模型 |
| 修改前创建备份 | 修改前备份到 `%APPDATA%/TieX/backups/` |
| 修改后显示 Diff | 文件修改后在 TaskDrawer 可查看 Diff |
| file_changes 正确保存 | 数据库 file_changes 表记录正确 |
| 可恢复单个文件 | 用户可点击"恢复"将文件还原 |
| 本任务授权不跨任务 | task 范围的授权不延续到新任务 |
| 停止任务后不再写入 | 任务停止后不会继续写入文件 |
| 哈希冲突时提示 | 恢复时文件哈希不匹配则提示冲突 |
| 审批记录持久化 | permission_requests 表正确记录每次审批 |
| 最多修改 20 个文件 | 超过任务上限时返回错误 |
