# 七、Provider 要求

## 7.1 ModelProvider 接口

必须定义统一的模型调用接口：

```ts
interface ModelProvider {
  testConnection(config: ProviderConfig): Promise<ProviderTestResult>;
  streamChat(request: ModelRequest, signal: AbortSignal): AsyncIterable<ModelStreamEvent>;
}
```

## 7.2 ProviderConfig

```ts
interface ProviderConfig {
  id: string;
  name: string;
  baseUrl: string;
  model: string;
  apiKey: string; // 解密后的明文，仅在主进程内存中使用
  temperature?: number;
  maxTokens?: number;
  timeoutMs: number;
}
```

注意：

- `apiKey` 仅在主进程内存中存在，不得写入日志、控制台或数据库；
- `apiKey` 从数据库读取时使用 `safeStorage.decryptString()` 解密；
- 每次请求从数据库读取并解密，不缓存明文 API Key。

## 7.3 ModelRequest

```ts
interface ModelRequest {
  messages: ChatMessage[];
  temperature?: number;
  maxTokens?: number;
}
```

## 7.4 ModelStreamEvent

```ts
type ModelStreamEvent =
  | { type: 'delta'; content: string }
  | { type: 'done'; usage?: TokenUsage }
  | { type: 'error'; error: ProviderError }
```

## 7.5 DeepSeek API 调用

- 使用 DeepSeek Chat Completions API：`POST {baseUrl}/chat/completions`
- 请求格式遵循 OpenAI 兼容协议；
- 使用 `stream: true` 启用流式输出；
- 使用 `fetch` 或 `node-fetch` 发起请求；
- 解析 SSE（Server-Sent Events）格式的响应；
- 逐个解析 `data: {...}` 行，提取 `choices[0].delta.content`；
- 遇到 `data: [DONE]` 时结束流。

## 7.6 请求取消

- 每次聊天请求创建独立的 `AbortController`；
- 用户点击"停止生成"时调用 `AbortController.abort()`；
- 停止后保留已生成的内容；
- 应用退出时取消所有进行中的请求。

## 7.7 错误分类

必须实现以下错误类型：

```text
PROVIDER_AUTH_ERROR       → API Key 无效或过期
PROVIDER_RATE_LIMIT       → 请求频率超限
PROVIDER_BALANCE_ERROR    → 账户余额不足
PROVIDER_TIMEOUT          → 请求超时
PROVIDER_NETWORK_ERROR    → 网络连接失败
PROVIDER_RESPONSE_INVALID → 响应格式异常
PROVIDER_NO_API_KEY       → 未配置 API Key
```

每种错误必须提供用户友好的中文提示。

# 八、聊天功能要求

## 8.1 消息发送流程

```text
用户输入消息
  ↓
前端发送 chat:send IPC
  ↓
主进程创建用户消息记录（写入 messages 表）
  ↓
主进程构建多轮上下文
  ↓
主进程创建 Assistant 消息记录（is_streaming=1）
  ↓
主进程调用 DeepSeekProvider.streamChat()
  ↓
每收到 delta 事件，更新 Assistant 消息内容
  ↓
每收到 delta 事件，通过 IPC 推送到前端
  ↓
流式完成，更新 is_streaming=0
  ↓
前端显示完整回复
```

## 8.2 多轮上下文

- 从数据库加载当前会话的所有历史消息；
- 按 `sequence_no` 排序；
- 构建符合 DeepSeek API 格式的 `messages` 数组；
- 上下文长度应有限制，超出时截断最早的消息；
- 系统提示词本阶段可简单固定，不动态构建。

## 8.3 流式消息更新

- 主进程每收到一个 delta，立即更新数据库中 Assistant 消息的 `content`；
- 同时通过 IPC 事件推送到前端；
- 前端实时更新消息气泡内容；
- 流式完成后标记 `is_streaming=0`。

## 8.4 停止生成

- 用户点击"停止"按钮；
- 前端发送 `chat:stop` IPC；
- 主进程调用 `AbortController.abort()`；
- 保留已生成的内容；
- 标记消息 `is_streaming=0`；
- 前端显示"已停止生成"提示。

## 8.5 连接测试

- 设置页的"测试连接"按钮改为真实调用 DeepSeek API；
- 发送一个简单的测试请求（如 `Hi`）；
- 成功返回"连接成功"；
- 失败返回具体错误信息（如"API Key 无效"、"网络连接失败"等）。

## 8.6 会话自动标题

- 用户发送第一条消息后，使用消息内容的前 30 个字符作为会话标题；
- 自动更新 `conversations.title`；
- 前端侧边栏同步更新。

# 九、IPC 要求

## 9.1 聊天相关 IPC

```text
chat:send              → 发送消息并开始流式回复
chat:stop              → 停止当前生成
chat:getMessages       → 获取会话消息列表
chat:onDelta           → 流式增量事件（主进程→渲染进程）
chat:onDone            → 流式完成事件（主进程→渲染进程）
chat:onError           → 流式错误事件（主进程→渲染进程）
```

## 9.2 IPC 事件推送

流式事件使用 `webContents.send()` 从主进程推送到渲染进程：

```ts
// 主进程推送
event.sender.send('chat:delta', { messageId, content, delta })
event.sender.send('chat:done', { messageId, usage })
event.sender.send('chat:error', { messageId, error })
```

```ts
// Preload 转发
chat: {
  onDelta: (callback) => {
    const handler = (_event, data) => callback(data)
    ipcRenderer.on('chat:delta', handler)
    return () => ipcRenderer.removeListener('chat:delta', handler)
  }
}
```

## 9.3 安全要求

- 渲染进程不得直接调用 DeepSeek API；
- 渲染进程不得接触 API Key 明文；
- 所有 API 请求由主进程发起；
- Preload 只暴露必要的 API；
- IPC 参数必须经过基本校验；
- `chat:send` 必须校验 `conversationId` 和 `content` 非空。
