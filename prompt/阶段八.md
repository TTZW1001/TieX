# 七、命令安全策略

## 7.1 CommandPolicy

CommandPolicy 负责判断命令是否允许执行。

### 允许的命令

默认允许以下命令（不限于，但需明确的允许规则）：

| 命令 | 用途 | 说明 |
|------|------|------|
| `npm test` | 运行测试 | 必须在工作区的 package.json 中有对应脚本 |
| `npm run build` | 运行构建 | 同上 |
| `npm run lint` | 代码检查 | 同上 |
| `npm install` | 安装依赖 | 仅限 `npm install`，不接受 `npm install -g` |
| `git status` | 查看 Git 状态 | 仅限工作区内的 Git 仓库 |
| `git diff` | 查看 Git 差异 | 同上 |
| `git log` | 查看 Git 日志 | 同上，最多 `--oneline -10` |
| `npx <工具>` | 运行本地工具 | 仅允许已在 package.json 中声明的工具 |
| `node <脚本>` | 运行 Node 脚本 | 脚本必须在工作区内 |
| `python <脚本>` | 运行 Python 脚本 | 脚本必须在工作区内 |

### 默认禁止的命令

| 类别 | 示例 |
|------|------|
| 删除目录树 | `rm -rf /`, `rd /s`, `del /f` |
| 系统命令 | `shutdown`, `regedit`, `taskkill`, `format` |
| 注册表操作 | `reg add`, `reg delete` |
| 关机重启 | `shutdown /s`, `restart-computer` |
| 编码 PowerShell | 含 `-EncodedCommand`, `-Encode`, `base64` 等 |
| 工作区外路径 | 命令参数中包含工作区外的路径 |
| 网络访问 | `curl`, `wget` 写系统目录 |
| 包全局安装 | `npm install -g`, `pip install --global` |
| 管道与重定向 | `\|`, `>`, `>>`, `<` (在大多数场景下) |
| 交互式命令 | 需要用户输入的交互式程序 |

### 策略接口

```ts
interface CommandPolicy {
  /**
   * 检查命令是否允许执行
   * 返回 { allowed: true } 或 { allowed: false, reason: string }
   */
  check(command: string, args: string[], cwd: string): CommandCheckResult

  /**
   * 判断是否危险命令（需审批）
   */
  isDangerous(command: string, args: string[]): boolean
}

interface CommandCheckResult {
  allowed: boolean
  reason?: string
  riskLevel?: 'medium' | 'high'
}
```

### 安全规则

- 命令和参数必须分离（模型提供命令 + 参数数组，不允许拼接字符串）；
- `cwd` 固定为工作区根目录，模型不可修改；
- 默认不使用 shell（`shell: false`），防止命令注入；
- 命令名称必须匹配允许列表中的可执行程序名；
- 参数中不能包含后门模式（如 `&&`, `;`, `|` 等 shell 元字符）；
- Windows 下检查 `.exe`, `.cmd`, `.ps1` 等扩展名的可执行文件路径。

## 7.2 风险等级

| 场景 | 风险等级 | 是否需要审批 |
|------|:--------:|:----------:|
| 执行 npm run build/test/lint | medium | 是 |
| 执行 npm install | medium | 是 |
| 执行 git 命令 | medium | 是 |
| 执行 npx 运行工具 | medium | 是 |
| 执行 node/python 脚本 | medium | 是 |
| 命令包含潜在危险参数 | high | 是 |
| 明确禁止的命令 | blocked | 否（直接拒绝） |

# 八、CommandRunner 要求

## 8.1 执行器设计

```ts
interface CommandRunner {
  /**
   * 执行命令，返回执行会话
   */
  run(input: RunCommandInput): Promise<CommandSession>

  /**
   * 通过会话 ID 停止命令
   */
  stop(sessionId: string): Promise<void>

  /**
   * 获取命令输出
   */
  getOutput(sessionId: string): CommandOutput

  /**
   * 停止所有正在运行的命令
   */
  stopAll(): Promise<void>
}

interface RunCommandInput {
  command: string         // 可执行程序名（如 npm, git, node）
  args: string[]          // 参数数组
  cwd: string             // 工作目录（由系统强制设为工作区根目录）
  timeoutMs: number       // 超时时间，默认 60000（1 分钟）
  maxOutputChars: number  // 最大输出字符数，默认 51200（50KB）
}

interface CommandSession {
  sessionId: string
  command: string
  args: string[]
  status: 'running' | 'completed' | 'failed' | 'stopped' | 'timeout'
  exitCode: number | null
  output: string
  truncated: boolean
  startedAt: string
  completedAt: string | null
}

interface CommandOutput {
  output: string
  truncated: boolean
  exitCode: number | null
}
```

## 8.2 执行流程

```text
接收 RunCommandInput
  ↓
检查命令和参数是否符合 CommandPolicy
  ├─ blocked → 直接返回错误（拒绝执行）
  └─ 允许 → 继续
  ↓
创建子进程（child_process.spawn）
  ├─ 不启用 shell（shell: false）
  ├─ cwd 固定为工作区
  ├─ 合并 stdout 和 stderr
  └─ 设置环境变量（过滤敏感变量）
  ↓
收集输出到缓冲区
  ↓
超过 maxOutputChars 时截断并标记 truncated
  ↓
超时 → kill 子进程树 → 标记 timeout
  ↓
用户停止 → kill 子进程树 → 标记 stopped
  ↓
进程退出 → 检查退出码 → 标记 completed / failed
  ↓
返回 CommandSession
```

## 8.3 进程管理

- 使用 `child_process.spawn()`，不使用 `exec()` 或 `execSync()`；
- `shell: false`（默认不启用 shell）；
- 记录进程 PID 到进程映射表；
- 停止时使用 `process.kill(pid, 'SIGTERM')` 后等待 3 秒，无响应则 `SIGKILL`；
- 跨平台进程树终止（Windows 下使用 `taskkill /T /F`）；
- 应用退出时遍历进程映射表终止所有子进程。

## 8.4 环境变量

默认环境变量继承父进程，但需过滤：

- 移除包含 `KEY`、`SECRET`、`TOKEN`、`PASSWORD`、`CREDENTIAL` 等敏感字样的变量；
- 保留 `PATH`、`HOME`、`USERPROFILE`、`APPDATA`、`NODE_ENV` 等必要变量；
- 工作区路径可注入到 `TIEX_WORKSPACE_ROOT` 环境变量供脚本使用。

## 8.5 超时

- 默认超时 60 秒（适用于 npm install 等耗时命令）；
- 超时后发送 SIGTERM；
- 3 秒后未退出则 SIGKILL；
- 标记状态为 `timeout`；
- 已收集的输出保留并返回。

## 8.6 输出管理

- 合并 stdout 和 stderr；
- 最大输出长度 50KB（超出截断并标记 `truncated: true`）；
- 流式推送输出到渲染进程（可选，便于展示实时输出）；
- 输出以 UTF-8 编码。

## 9.5 规则

- 命令执行失败（exitCode !== 0）时，输出仍返回给模型；
- 不允许同时运行多条命令（单条命令执行完毕后再执行下一条）；

      risk: 'medium',
      reason: '将执行项目构建/测试命令'
    },
    {
      condition: (input) => true,

# 十一、IPC 与事件要求

## 11.1 IPC 通道

```text
command:run      → 执行命令（仅内部，不暴露给渲染进程直接调用）
command:stop     → 停止正在执行的命令
command:getOutput → 获取命令当前输出
```

命令执行由 Agent Runtime 通过 Tool Executor 触发，不通过渲染进程直接调用 IPC。渲染进程通过任务事件接收命令状态更新。

## 11.2 事件推送

新增命令执行事件类型：

```ts
// 命令状态变更
{ type: 'command:output', taskId, sessionId, output: string, truncated: boolean }
{ type: 'command:completed', taskId, sessionId, exitCode, output }
{ type: 'command:failed', taskId, sessionId, error }
{ type: 'command:stopped', taskId, sessionId }
{ type: 'command:timeout', taskId, sessionId }
```

这些事件通过现有的 Task Event Bus 推送。

# 十六、应用退出处理

在 `electron/main/index.ts` 中扩展退出处理逻辑：

```typescript
// 应用退出时
app.on('will-quit', async () => {
  // 1. 停止所有 running 任务（已有）
  // 2. 终止所有子进程（新增）
  await commandRunner.stopAll()
})
```

停止所有子进程的流程：

1. 遍历进程映射表；
2. 对每个进程发送 SIGTERM（Windows 下使用 `taskkill /T /F`）；
3. 最多等待 5 秒；
4. 未退出进程发送 SIGKILL。
